      <h2>API Documentation</h2>
      <p><b>API Base URL:</b>
        https://clinicaltables.nlm.nih.gov/api/icd10cm/v3/search
        (+ <a href="#params">query string parameters</a>)</p>

            <p>
        This data set may also be accessed through the <a href="/valueSetExpand.html">FHIR ValueSet $expand operation</a>.
      </p>
      <p>In addition to the base URL, you will need to specify other parameters.
        See the <a href="#params">query string parameters</a> section below for details.



            <h3><a name="params"></a>Query String Parameters and Default Values</h3>
      <p>At a minimum, when using the above base URL, you will need to specify
        the "terms" parameter containing a word or partial word to match.</p>
      <table>
          <thead><tr><th>Parameter Name</th><th>Default Value</th><th>Description</th></tr></thead>
        <tbody>
                    <tr><td>terms</td><td></td><td>(Required.)  The search string (e.g.,
            just a part of a word) for which to find matches in the list.  More than one
            partial word can be present in "terms", in which case there is an implicit AND
            between them.</td></tr>


          <tr><td>maxList</td><td>7</td><td>
            Optional, with a default of 7. Specifies the number of results requested, up to the upper
            limit of 500. If present but the value is empty, 500 will be used. Note that this parameter
            does not support pagination, see "count" and "offset" below for details on pagination support.
          </td></tr>
          <tr><td>count</td><td>7</td>
            <td>The number of results to retrieve (page size). The maximum count allowed is 500,
              see "offset" below on pagination support.</td>
          </tr>
          <tr><td>offset</td><td>0</td>
            <td>The starting result number (0-based) to retrieve. Use offset and count together for pagination.
              Note that the current limit on the total number of results that can be retrieved (offset + count)
              is 7,500. We reserve the right to decrease or increase this limit based on system capacity and/or
              other factors. Please see the <a href="/faq.html"><b>FAQ page</b></a> on how to sign up to our
              email list to be notified of any changes or new features.
            </td>
          </tr>
          <tr><td>q</td><td></td><td>An optional, additional query string
            used to further constrain the results returned by the "terms" field.
            Unlike the terms field, "q" is not automatically wildcarded, but
            can include wildcards and can specify field names.  See the Elasticsearch
            <a href="https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-query-string-query.html#query-string-syntax">query
            string</a> page for documentation of supported syntax.</td></tr>

          <tr><td>df</td><td>code, name</td><td>A comma-separated list of display
fields (from the <a href="#fields">fields</a> section below) which are
intended for the user to see when looking at the results.

            </td></tr>
          <tr><td>sf</td><td>code</td><td>A comma-separated list of fields to be searched.
                 </td></tr>
          <tr><td>cf</td><td>code</td><td>A field to regard as the "code" for the returned item
                  data.</td></tr>
          <tr><td>ef</td><td></td><td>A comma-separated list of additional fields to be
returned for each retrieved list item.  (See the <a
href="#output">Output format</a> section for how the data for fields
is returned.)  If you wish the keys in the returned data hash to be something
other than the field names, you can specify an alias for the field name by
separating it from its field name with a colon, e.g.,
"ef=field_name1:alias1,field2,field_name3:alias3,etc.  Note that not
every field specified in the ef parameter needs to have an alias.

            </td></tr>
        </tbody>
      </table>

      <h3><a name="fields"></a>ICD-10-CM Field Descriptions</h3>
      <table>
        <thead>
            <tr><th>Field</th><th>Field Description</th></tr>
        </thead>
        <tbody>
            <tr><td>code</td><td>The ICD-10-CM unique disease code containing 3, 4, 5, 6 or 7 digits.</td></tr>
<tr><td>name</td><td>The long description of the diagnosis string(s).</td></tr>

        </tbody>
      </table>

            <h3><a name="output"></a>Output format</h3>
      <p>Output for an API query is an array of the following elements:</p>
      <ol>
        <li>The total number of results on the server, which can be more than the
          number of results returned. This reported total number of results may also be
          significantly less than the actual number of results and is limited to
          10,000, which may significantly improve the service response time.</li>
        <li>An array of codes for the returned items.  (This is the field
  specified with the cf query parameter above.)</li>
        <li>A hash of the "extra" data requested via the "ef" query parameter
  above.  The keys on the hash are the fields (or their requested aliases) named
  in the "ef" parameter, and the value for a field is an array of that field's
  values in the same order as the returned codes.</li>
        <li>An array, with one element for each returned code, where each element
  is an array of the display strings specified with the "df" query parameter.</li>
        <li>An array, with one element for each returned code, where each element is the "code system" for
  the returned code. Note that only code-system aware APIs will return this array.</li>
      </ol>
