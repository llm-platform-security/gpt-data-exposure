            <div class="main-content lit-style" itemscope="itemscope" itemtype="http://schema.org/CreativeWork"><div class="meta-content fm-sec"><h1 id="_NBK25497_"><span class="title" itemprop="name">A General Introduction to the E-utilities</span></h1><p class="contrib-group"><span itemprop="author">Eric Sayers</span>, PhD.</p><a data-jig="ncbitoggler" href="#__NBK25497_ai__" style="border:0;text-decoration:none">Author Information and Affiliations</a><div style="display:none" class="ui-widget" id="__NBK25497_ai__"><p class="contrib-group"><h4>Authors</h4><span itemprop="author">Eric Sayers</span>, PhD<sup><img src="/corehtml/pmc/pmcgifs/corrauth.gif" alt="corresponding author" /></sup><sup>1</sup>.</p><h4>Affiliations</h4><div class="affiliation"><sup>1</sup> NCBI<div><span class="email-label">Email: </span><a href="mailto:dev@null" data-email="vog.hin.mln.ibcn@sreyas" class="oemail">vog.hin.mln.ibcn@sreyas</a></div></div><div><sup><img src="/corehtml/pmc/pmcgifs/corrauth.gif" alt="corresponding author" /></sup>Corresponding author.</div></div><p class="small">Created: <span itemprop="datePublished">May 26, 2009</span>; Last Update: <span itemprop="dateModified">November 17, 2022</span>.</p><p><em>Estimated reading time: 11 minutes</em></p></div><div class="jig-ncbiinpagenav body-content whole_rhythm" data-jigconfig="allHeadingLevels: ['h2'],smoothScroll: false" itemprop="text"><div id="chapter2.Introduction"><h2 id="_chapter2_Introduction_">Introduction</h2><p>The Entrez Programming Utilities (E-utilities) are a set of nine server-side programs that provide a stable interface into the Entrez query and database system at the National Center for Biotechnology Information (NCBI). The E-utilities use a fixed URL syntax that translates a standard set of input parameters into the values necessary for various NCBI software components to search for and retrieve the requested data. The E-utilities are therefore the structured interface to the Entrez system, which currently includes 38 databases covering a variety of biomedical data, including nucleotide and protein sequences, gene records, three-dimensional molecular structures, and the biomedical literature.</p><p>To access these data, a piece of software first posts an E-utility URL to NCBI, then retrieves the results of this posting, after which it processes the data as required. The software can thus use any computer language that can send a URL to the E-utilities server and interpret the XML response; examples of such languages are Perl, Python, Java, and C++. Combining E-utilities components to form customized data pipelines within these applications is a powerful approach to data manipulation.</p><p>This chapter first describes the general function and use of the eight E-utilities, followed by basic usage guidelines and requirements, and concludes with a discussion of how the E-utilities function within the Entrez system.</p></div><div id="chapter2.Usage_Guidelines_and_Requiremen"><h2 id="_chapter2_Usage_Guidelines_and_Requiremen_">Usage Guidelines and Requirements</h2><div id="chapter2.Use_the_Eutility_URL"><h3>Use the E-utility URL</h3><p>All E-utility requests should be made to URLs beginning with the following string:</p><p>
<a href="https://eutils.ncbi.nlm.nih.gov/entrez/eutils/" ref="pagearea=body&amp;targetsite=external&amp;targetcat=link&amp;targettype=uri">https://eutils.ncbi.nlm.nih.gov/entrez/eutils/</a>
</p><p>These URLs direct requests to servers that are used only by the E-utilities and that are optimized to give users the best performance.</p></div><div id="chapter2.Frequency_Timing_and_Registrati"><h3>Frequency, Timing and Registration of E-utility URL Requests</h3><p>In order not to overload the E-utility servers, NCBI recommends that users post no more than three URL requests per second and limit large jobs to either weekends or between 9:00 PM and 5:00 AM Eastern time during weekdays. Failure to comply with this policy may result in an IP address being blocked from accessing NCBI. If NCBI blocks an IP address, service will not be restored unless the developers of the software accessing the E-utilities register values of the <b>tool</b> and <b>email</b> parameters with NCBI. The value of <b>tool</b> should be a string with no internal spaces that uniquely identifies the software producing the request. The value of <b>email</b> should be a complete and valid e-mail address of the software developer and not that of a third-party end user. The value of <b>email</b> will be used only to contact developers if NCBI observes requests that violate our policies, and we will attempt such contact prior to blocking access. In addition, developers may request that the value of <b>email</b> be added to the E-utility mailing list that provides announcements of software updates, known bugs and other policy changes affecting the E-utilities. To register <b>tool</b> and <b>email</b> values, simply send an e-mail to <a href="mailto:dev@null" data-email="vog.hin.mln.ibcn@seitilitue" class="oemail">vog.hin.mln.ibcn@seitilitue</a> including the desired values along with the name of either a developer or the organization creating the software. Once NCBI establishes communication with a developer, receives values for <b>tool</b> and <b>email</b> and validates the e-mail address in <b>email</b>, the block will be lifted. Once <b>tool</b> and <b>email</b> values are registered, all subsequent E-utility requests from that software package should contain both values. Please be aware that merely providing values for <b>tool</b> and <b>email</b> in requests is not sufficient to comply with this policy; these values must be registered with NCBI. Requests from any IP that lack registered values for <b>tool</b> and <b>email</b> and that violate the above usage policies may be blocked. Software developers may register values of <b>tool</b> and <b>email</b> at any time, and are encouraged to do so.</p></div><div id="chapter2.API_Keys"><h3>API Keys</h3><p>Since December 1, 2018, NCBI has provided API keys that offer enhanced levels of supported access to the E-utilities. Without an API key, any site (IP address) posting more than 3 requests per second to the E-utilities will receive an error message. By including an API key, a site can post up to 10 requests per second by default. Higher rates are available by request (<a href="mailto:dev@null" data-email="vog.hin.mln.ibcn@seitilitue" class="oemail">vog.hin.mln.ibcn@seitilitue</a>). Users can obtain an API key now from the Settings page of their NCBI account (to create an account, visit <a href="/account/" ref="pagearea=body&amp;targetsite=external&amp;targetcat=link&amp;targettype=uri">http://www.ncbi.nlm.nih.gov/account/</a>). After creating the key, users should include it in each E-utility request by assigning it to the <i>api_key</i> parameter.</p><pre><b>Example request including an API key:</b><br />esummary.fcgi?db=pubmed&#x00026;id=123456&#x00026;api_key=ABCDE12345<br /><br /><b>Example error message if rates are exceeded:</b><br />{"error":"API rate limit exceeded","count":"11"}</pre><p>Only one API key is allowed per NCBI account; however, a user may request a new key at any time. Such a request will invalidate any existing API key associated with that NCBI account.</p></div><div id="chapter2.Minimizing_the_Number_of_Reques"><h3>Minimizing the Number of Requests</h3><p>If a task requires searching for and/or downloading a large number of records, it is much more efficient to use the Entrez History to upload and/or retrieve these records in batches rather than using separate requests for each record. Please refer to <a href="/books/n/helpeutils/chapter3/#chapter3.Application_3_Retrieving_large">Application 3 in Chapter 3</a> for an example. Many thousands of IDs can be uploaded using a single EPost request, and several hundred records can be downloaded using one EFetch request.</p></div><div id="chapter2.Disclaimer_and_Copyright_Issues"><h3>Disclaimer and Copyright Issues</h3><p>If you use the E-utilities within software, NCBI's Disclaimer and Copyright notice (<a href="https://www.ncbi.nlm.nih.gov/About/disclaimer.html" ref="pagearea=body&amp;targetsite=external&amp;targetcat=link&amp;targettype=uri">https://www.ncbi.nlm.nih.gov/About/disclaimer.html</a>) must be evident to users of your product. There is no required order for the URL parameters in an E-utility URL, and null values or inappropriate parameters are generally ignored. Avoid placing spaces in the URLs, particularly in queries. If a space is required, use a plus sign (+) instead of a space:</p><pre><b>Incorrect</b>: &#x00026;id=352, 25125, 234<br /><b>Correct</b>:   &#x00026;id=352,25125,234<br /><br /><b>Incorrect</b>: &#x00026;term=biomol mrna[properties] AND mouse[organism]<br /><b>Correct</b>:   &#x00026;term=biomol+mrna[properties]+AND+mouse[organism]</pre><p>Other special characters, such as quotation marks (&#x0201c;) or the # symbol used in referring to a query key on the History server, should be represented by their URL encodings (%22 for &#x0201c;; %23 for #).</p><pre><b>Incorrect</b>: &#x00026;term=#2+AND+"gene in genomic"[properties]<br /><b>Correct</b>:   &#x00026;term=%232+AND+%22gene+in+genomic%22[properties]</pre></div></div><div id="chapter2.The_Nine_Eutilities_in_Brief"><h2 id="_chapter2_The_Nine_Eutilities_in_Brief_">The Nine E-utilities in Brief</h2><div id="chapter2.EInfo_database_statistics"><h3>EInfo (database statistics)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/einfo.fcgi</i>
</p><p>Provides the number of records indexed in each field of a given database, the date of the last update of the database, and the available links from the database to other Entrez databases.</p></div><div id="chapter2.ESearch_text_searches"><h3>ESearch (text searches)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/esearch.fcgi</i>
</p><p>Responds to a text query with the list of matching UIDs in a given database (for later use in ESummary, EFetch or ELink), along with the term translations of the query.</p></div><div id="chapter2.EPost_UID_uploads"><h3>EPost (UID uploads)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/epost.fcgi</i>
</p><p>Accepts a list of UIDs from a given database, stores the set on the History Server, and responds with a query key and web environment for the uploaded dataset.</p></div><div id="chapter2.ESummary_document_summary_downl"><h3>ESummary (document summary downloads)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/esummary.fcgi</i>
</p><p>Responds to a list of UIDs from a given database with the corresponding document summaries.</p></div><div id="chapter2.EFetch_data_record_downloads"><h3>EFetch (data record downloads)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/efetch.fcgi</i>
</p><p>Responds to a list of UIDs in a given database with the corresponding data records in a specified format.</p></div><div id="chapter2.ELink_Entrez_links"><h3>ELink (Entrez links)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/elink.fcgi</i>
</p><p>Responds to a list of UIDs in a given database with either a list of related UIDs (and relevancy scores) in the same database or a list of linked UIDs in another Entrez database; checks for the existence of a specified link from a list of one or more UIDs; creates a hyperlink to the primary LinkOut provider for a specific UID and database, or lists LinkOut URLs and attributes for multiple UIDs.</p></div><div id="chapter2.EGQuery_global_query"><h3>EGQuery (global query)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/egquery.fcgi</i>
</p><p>Responds to a text query with the number of records matching the query in each Entrez database.</p></div><div id="chapter2.ESpell_spelling_suggestions"><h3>ESpell (spelling suggestions)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/espell.fcgi</i>
</p><p>Retrieves spelling suggestions for a text query in a given database.</p></div><div id="chapter2.ECitMatch_batch_citation_search"><h3>ECitMatch (batch citation searching in PubMed)</h3><p>
<i>eutils.ncbi.nlm.nih.gov/entrez/eutils/ecitmatch.cgi</i>
</p><p>Retrieves PubMed IDs (PMIDs) corresponding to a set of input citation strings.</p></div></div><div id="chapter2.Understanding_the_Eutilities_Wi"><h2 id="_chapter2_Understanding_the_Eutilities_Wi_">Understanding the E-utilities Within Entrez</h2><div id="chapter2.The_Eutilities_Access_Entrez_Da"><h3>The E-utilities Access Entrez Databases</h3><p>The E-utilities access the core search and retrieval engine of the Entrez system and, therefore, are only capable of retrieving data that are already in Entrez. Although the majority of data at NCBI are in Entrez, there are several datasets that exist outside of the Entrez system. Before beginning a project with the E-utilities, check that the desired data can be found within an Entrez database.</p></div><div id="chapter2.The_Entrez_System_Identifies_Da"><h3>The Entrez System Identifies Database Records Using UIDs</h3><p>Each Entrez database refers to the data records within it by an integer ID called a UID (unique identifier). Examples of UIDs are GI numbers for Nucleotide and Protein, PMIDs for PubMed, or MMDB-IDs for Structure. The E-utilities use UIDs for both data input and output, and thus it is often critical, especially for advanced data pipelines, to know how to find the UIDs associated with the desired data before beginning a project with the E-utilities.</p><p>See <a href="/books/NBK25497/table/chapter2.T._entrez_unique_identifiers_ui/?report=objectonly" target="object" rid-ob="figobchapter2Tentrezuniqueidentifiersui">Table 1</a> for a complete list of UIDs in Entrez.</p><div class="iconblock whole_rhythm clearfix ten_col table-wrap" id="figchapter2Tentrezuniqueidentifiersui"><a href="/books/NBK25497/table/chapter2.T._entrez_unique_identifiers_ui/?report=objectonly" target="object" title="Table 1 " class="img_link icnblk_img" rid-ob="figobchapter2Tentrezuniqueidentifiersui"><img class="small-thumb" src="/corehtml/pmc/css/bookshelf/2.26/img/table-icon.gif" alt="Table Icon" /></a><div class="icnblk_cntnt"><h4 id="chapter2.T._entrez_unique_identifiers_ui"><a href="/books/NBK25497/table/chapter2.T._entrez_unique_identifiers_ui/?report=objectonly" target="object" rid-ob="figobchapter2Tentrezuniqueidentifiersui">Table 1 </a></h4><p class="float-caption no_bottom_margin">&#x02013; Entrez Unique Identifiers (UIDs) for selected databases </p></div></div></div><div id="chapter2.Accessing_Sequence_Records_Usin"><h3>Accessing Sequence Records Using Accession.Version Identifiers</h3><p>NCBI now uses the accession.version identifier rather that the GI number (UID) as the primary identifier for nucleotide and protein sequence records (records in the nuccore, nucest, nucgss, popset, and protein databases). Even so, the E-utilities continue to provide access to these records using either GI numbers or accession.version identifiers. Those E-utilities that accept UIDs as input will also accept accession.version identifiers (for the sequence databases listed above). Those E-utilities that output UIDs can output accession.version identifiers instead by setting the &#x00026;idtype parameter to &#x0201c;acc&#x0201d;. Finally, EFetch can retrieve <i>any</i> sequence record by its accession.version identifier, including sequences that do not have GI numbers. Please see <a href="/books/n/helpeutils/chapter4/">Chapter 4</a> for more details about how each E-utility handles accession.version identifers.</p></div><div id="chapter2.The_Entrez_Core_Engine_EGQuery"><h3>The Entrez Core Engine: EGQuery, ESearch, and ESummary</h3><p>The core of Entrez is an engine that performs two basic tasks for any Entrez database: 1) assemble a list of UIDs that match a text query, and 2) retrieve a brief summary record called a Document Summary (DocSum) for each UID. These two basic tasks of the Entrez engine are performed by ESearch and ESummary. ESearch returns a list of UIDs that match a text query in a given Entrez database, and ESummary returns DocSums that match a list of input UIDs. A text search in web Entrez is equivalent to ESearch-ESummary. EGQuery is a global version of ESearch that searches all Entrez databases simultaneously. Because these three E-utilities perform the two core Entrez functions, they function for all Entrez databases.</p><pre>egquery.fcgi?term=<i>query</i><br />esearch.fcgi?db=<i>database</i>&#x00026;term=<i>query</i><br />esummary.fcgi?db=<i>database</i>&#x00026;id=<i>uid1,uid2,uid3,...</i></pre></div><div id="chapter2.Syntax_and_Initial_Parsing_of_E"><h3>Syntax and Initial Parsing of Entrez Queries</h3><p>Text search strings entered into the Entrez system are converted into Entrez queries with the following format:</p><p>term1[field1] <b>Op</b> term2[field2] <b>Op</b> term3[field3] <b>Op</b> ...</p><p>where the terms are search terms, each limited to a particular Entrez field in square brackets, combined using one of three Boolean operators: Op = AND, OR, or NOT. These Boolean operators must be typed in all capital letters.</p><p>Example: human[organism] AND topoisomerase[protein name]</p><p>Entrez initially splits the query into a series of items that were originally separated by spaces in the query; therefore it is critical that spaces separate each term and Boolean operator. If the query consists <i>only</i> of a list of UID numbers (unique identifiers) or accession numbers, the Entrez system simply returns the corresponding records and no further parsing is performed. If the query contains any Boolean operators (AND, OR, or NOT), the query is split into the terms separated by these operators, and then each term is parsed independently. The results of these searches are then combined according to the Boolean operators.</p><p>A full account of how to search Entrez can be found in the <a href="/books/n/helpentrez/EntrezHelp/">Entrez Help Document</a>. Additional information is available from <a href="/books/n/helpentrez/">Entrez Help</a>.</p></div><div id="chapter2.Entrez_Databases_EInfo_EFetch_a"><h3>Entrez Databases: EInfo, EFetch, and ELink</h3><p>The NCBI Entrez system currently contains 38 databases. EInfo provides detailed information about each database, including lists of the indexing fields in the database and the available links to other Entrez databases.</p><pre>einfo.fcgi?db=<i>database</i></pre><p>Each Entrez database includes two primary enhancements to the raw data records: 1) software for producing a variety of display formats appropriate to the given database, and 2) links to records in other Entrez databases manifested as lists of associated UIDs. The display format function is performed by EFetch, which generates formatted output for a list of input UIDs. For example, EFetch can produce abstracts from Entrez PubMed or FASTA format from Entrez Protein. EFetch does not yet support all Entrez databases; please see the <a href="/books/n/helpeutils/chapter4/#chapter4.EFetch">EFetch documentation</a> for details.</p><pre>efetch.fcgi?db=database&#x00026;id=<i>uid1,uid2,uid3</i>&#x00026;rettype=<i>report_type</i>&#x00026;retmode=<br /><i>data_mode</i></pre><p>The linking function is performed by ELink, which generates a list of UIDs in a specified Entrez database that are linked to a set of input UIDs in either the same or another database. For example, ELink can find Entrez SNP records linked to records in Entrez Nucleotide, or Entrez Domain records linked to records in Entrez Protein.</p><pre>elink.fcgi?dbfrom=<i>initial_database</i>db=<i>target_database</i>&#x00026;id=<i>uid1,uid2,uid3</i></pre></div><div id="chapter2.Using_the_Entrez_History_Server"><h3>Using the Entrez History Server</h3><p>A powerful feature of the Entrez system is that it can store retrieved sets of UIDs temporarily on the servers so that they can be subsequently combined or provided as input for other E-utility calls. The Entrez History server provides this service and is accessed on the Web using either the Preview/Index or History tabs on Entrez search pages. Each of the E-utilities can also use the History server, which assigns each set of UIDs an integer label called a query key (&#x00026;query_key) and an encoded cookie string called a Web environment (&#x00026;WebEnv). EPost allows any list of UIDs to be uploaded to the History Server and returns the query key and Web environment. ESearch can also post its output set of UIDs to the History Server, but only if the &#x00026;usehistory parameter is set to &#x0201c;y&#x0201d;. ELink also can post its output to the History server if &#x00026;cmd is set to "neighbor_history". The resulting query key and Web environment from either EPost or ESearch can then be used in place of a UID list in ESummary, EFetch, and ELink.</p><p>In Entrez, a set of UIDs is represented on the History by three parameters:</p><pre>&#x00026;db = database; &#x00026;query_key = query key; &#x00026;WebEnv = web environment</pre><p>Upload steps that generate a web environment and query key</p><pre>esearch.fcgi?db=<i>database</i>&#x00026;term=<i>query</i><b>&#x00026;usehistory=y</b><br /><br />epost.fcgi?db=<i>database</i>&#x00026;id=<i>uid1,uid2,uid3,...</i><br /><br />elink.fcgi?dbfrom=<i>source_db</i>&#x00026;db=<i>destination_db</i>&#x00026;cmd=neighbor_history&#x00026;id=<br /><i>uid1,uid2,...</i></pre><p>Download steps that use a web environment and query key</p><pre>esummary.fcgi?db=<i>database</i>&#x00026;WebEnv=<i>webenv</i>&#x00026;query_key=<i>key</i><br /><br />efetch.fcgi?db=<i>database</i>&#x00026;WebEnv=<i>webenv</i>&#x00026;query_key=<i>key</i>&#x00026;rettype=<br /><i>report_type</i>&#x00026;retmode=<i>data_mode</i></pre><p>Link step that uses a web environment and query key</p><pre>elink.fcgi?dbfrom=<i>initial_database</i>db=<i>target_database</i>&#x00026;WebEnv=<br /><i>webenv</i>&#x00026;query_key=<i>key</i></pre><p>Search step that uses a web environment and a query key in the &#x00026;term parameter (preceded by #, encoded as %23)</p><pre>esearch.fcgi?db=<i>database</i>&#x00026;term=<i>%23key+AND+query</i>&#x00026;WebEnv=<i>webenv</i>&#x00026;usehistory=y</pre></div><div id="chapter2.Generating_Multiple_Data_Sets_o"><h3>Generating Multiple Data Sets on the History Server</h3><p>Each web environment on the History Server can be associated with any number of query keys. This allows different data sets to be combined with the Boolean operators AND, OR, and NOT, or with another Entrez query. It is important to remember that for two data sets (query keys) to be combined, they must be associated with the same web environment. By default, successive E-utility calls produce query keys that are <i>not</i> associated with the same web environment, and so to overcome this, each E-utility call after the initial call must set the &#x00026;WebEnv parameter to the value of the pre-existing web environment.</p><p>Default behavior: These two URLs&#x02026;</p><pre>URL 1: epost.fcgi?db=<i>database</i>&#x00026;id=<i>uid1,uid2,uid3</i><br />URL 2: esearch.fcgi?db=<i>database</i>&#x00026;term=<i>query</i>&#x00026;usehistory=y</pre><p>will produce two History sets associated with different web environments:</p><pre><b>URL   WebEnv     query_key        UIDs</b><br />1      web1          1        uid1,uid2,uid3<br />2      web2          1        uids matching query</pre><p>Desired behavior: These two URLs&#x02026;</p><pre>URL 1: epost.fcgi?db=<i>database</i>&#x00026;id=<i>uid1,uid2,uid3</i><br />(extract <i>web1</i> from the output of URL 1)<br />URL 2: esearch.fcgi?db=<i>database</i>&#x00026;term=<i>query</i>&#x00026;usehistory=y<b>&#x00026;WebEnv=<i>web1</i></b></pre><p>will produce two sets associated with the same (new) web environment:</p><pre><b>URL   WebEnv      query_key        UIDs</b><br />1      web2          1        uid1,uid2,uid3<br />2      web2          2        uids matching query</pre></div></div><div id="chapter2.Combining_Eutility_Calls_to_Cre"><h2 id="_chapter2_Combining_Eutility_Calls_to_Cre_">Combining E-utility Calls to Create Entrez Applications</h2><p>The E-utilities are useful when used by themselves in single URLs; however, their full potential is realized when successive E-utility URLs are combined to create a data pipeline.
